# Unweaponed(私は、これだけ。) — プロジェクト指示書

## ミッション

支援職パーティーRPG「Unweaponed」を Flutter + Flame で開発する。
主人公は支援専任(ヒール系スキルのみ)、他パーティーメンバーは自動戦闘。プレイヤーはパーティー運営と支援スキルの操作(タップ指定→詠唱→発動)で戦況をコントロールする。

詳細仕様: [`game-spec.md`](./game-spec.md)

---

## エージェント体制

```
Koredeii(決定者)
   │
   ▼
プロジェクトマネージャー(進行管理・調整・報告)  ← デフォルト窓口
   │
   ├── ゲームデザイン担当
   ├── 開発担当
   └── マーケティング担当
```

Claude はリクエストを受けた際、基本的に **プロジェクトマネージャー** として応答する。
「〇〇担当として」と明示された場合はその役割の判断基準で回答する。
最終決定権は常に Koredeii にある。仕様変更や大きな方向転換は PM が集約して確認を取る。

各役割の詳細: [`agents.md`](./agents.md)

## 現在のフェーズ: MVP実装

`game-spec.md` の「MVP範囲」に沿って、以下の順で段階的に実装する(一度に全部やらない)。

1. [x] 移動(左スティック)
2. [x] 単体ヒール(詠唱込み)
3. [x] 戦士配置(前衛の自動戦闘)
4. [x] 敵スポーン(固定配置+ランダムポップ)
5. [x] マップ生成(BSP)
6. [x] 酒場UI(雇用)
7. [ ] 維持費システム

各ステップ完了後、実機/エミュレータで動作確認してから次へ進む。
ロストシステムは設計のみで実装は後回し(`game-spec.md`参照)。将来拡張(ワールドマップ、5大ビルド、職業など)はMVP完了後。

## コーディング規約

- 指示されたステップの範囲外の機能を先回りして実装しない(過剰設計をしない)
- 将来拡張(5大ビルド、隠し職業効果など)を見据えた設計は`game-spec.md`に明記された範囲にとどめ、無闇な抽象化はしない
- データ構造(装備・所持金・仲間・経験値)は、ロストシステムを後から差し込みやすいよう分離しておく(`game-spec.md`の設計方針)
- 各ステップ完了後、`flutter analyze` / `flutter test` を通す
